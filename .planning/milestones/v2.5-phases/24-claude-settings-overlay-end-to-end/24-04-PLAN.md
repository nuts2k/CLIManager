在后端完成 apply 触发链路，并明确落地本 Plan 选定的启动结果交付策略：`startup 阶段不依赖实时 emit，而是写入缓存队列，由前端稍后主动读取`。

1) 在 `src-tauri/src/commands/claude_settings.rs` 中统一定义 overlay apply 通知模型（用于实时事件与 startup 缓存共用）：
- 事件名保持三类：
  - `claude-overlay-apply-success`
  - `claude-overlay-apply-failed`
  - `claude-overlay-protected-fields-ignored`
- 新增统一 payload 结构，例如：
  - `kind: "success" | "failed" | "protected_fields_ignored"`
  - `source: "save" | "startup" | "watcher"`
  - `settings_path?: string`
  - `error?: string`
  - `paths?: string[]`

2) 新增 startup 通知缓存队列并注册到 app state：
- 在 `src-tauri/src/lib.rs` 增加类似 `ClaudeOverlayStartupNotificationQueue(Mutex<Vec<ClaudeOverlayApplyNotification>>)` 的 state
- `.setup()` 前通过 `.manage(...)` 注册该 state
- 该队列只负责缓存 `source="startup"` 期间产生的通知；save / watcher 不写这里

3) 新增 `#[tauri::command] pub fn take_claude_overlay_startup_notifications(...) -> Result<Vec<ClaudeOverlayApplyNotification>, AppError>`：
- 一次性返回并清空 startup 队列（take 语义）
- 前端启动后只调用这一条命令获取 startup 期间积累的 success / failed / protected-fields-ignored 通知
- 不要新增“peek 但不清空”方案，避免重复 toast

4) 完成 `apply_claude_settings_overlay(app_handle, source)`：
- 读取当前 local settings -> 获取 claude active provider
  - 有活跃 provider：使用 ClaudeAdapter.patch(provider)，确保保护字段最终仍由 Provider/Proxy 决定
  - 无活跃 provider：仍允许 apply overlay，但必须先 strip overlay 中保护字段，不能把错误 token/base_url 写进 `~/.claude/settings.json`
- 读取 overlay 文件：
  - `None`：视为 noop（best-effort）
  - `Some(text)`：必须是合法 JSON 且 root 为 object，否则返回 `AppError::Validation(...)`
- overlay 中包含保护字段时，生成 `protected_fields_ignored` 通知
- `~/.claude/settings.json` 存在但非法 JSON 时，保持现有策略：拒绝写入并返回 Validation

5) 按 source 分流通知发送方式：
- `source="save"` / `source="watcher"`：实时 `emit` 到前端
- `source="startup"`：不要依赖实时 emit；把 success / failed / protected_fields_ignored 通知追加到 startup 队列，并同时写日志
- 这样启动结果的 UI 可见性只依赖缓存 + take 命令，不依赖 setup 时序

6) 修改 `set_claude_settings_overlay`：
- 保存成功后立即调用 `apply_claude_settings_overlay(source="save")`
- 若 apply 失败：
  - set 命令整体返回 Err（强一致，COVL-09）
  - 同时实时 emit failed 事件，供当前前端会话立即看到错误

7) 在 `src-tauri/src/lib.rs` 的 `.setup()` 阶段：
- spawn 一个 async 任务调用 `apply_claude_settings_overlay(source="startup")`
- 失败只写 startup 队列 + log，不阻断启动（COVL-10）

8) 注册命令：
- `commands::claude_settings::apply_claude_settings_overlay`
- `commands::claude_settings::take_claude_overlay_startup_notifications`

注意：
- 这一步必须把“startup 结果缓存后再由前端主动获取”的策略写死到实现任务里，不能再保留“直接 emit 也许能收到”的模糊方案
- save 触发保持强一致；startup / watcher 保持 best-effort
