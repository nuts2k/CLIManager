Four wiring changes across Rust and frontend:

**1. src-tauri/src/watcher/mod.rs -- Add tray rebuild after iCloud sync**

In the `process_events` function, add a tray menu rebuild call AFTER the existing `app_handle.emit("providers-changed", &payload)` call (around line 69). Add this line:

```rust
// Rebuild tray menu to reflect provider changes from iCloud sync
#[cfg(desktop)]
crate::tray::update_tray_menu(app_handle);
```

The `#[cfg(desktop)]` guard matches the existing `tray.rs` module guard. This ensures iCloud sync events (provider add/edit/delete from another device) immediately update the tray menu.

**2. src/lib/tauri.ts -- Add refreshTrayMenu invoke wrapper**

Add at the end of the file:

```typescript
export async function refreshTrayMenu(): Promise<void> {
  return invoke("refresh_tray_menu");
}
```

**3. src/components/settings/SettingsPage.tsx -- Call refreshTrayMenu after language change**

Import `refreshTrayMenu` from `@/lib/tauri` at the top of the file.

Update `handleLanguageChange` to:
```typescript
const handleLanguageChange = async (lang: string) => {
    await i18n.changeLanguage(lang);
    await updateSettings({ language: lang });
    await refreshTrayMenu();
};
```

This ensures that when the user switches language in settings, the tray menu labels update immediately (per MENU-03 and CONTEXT.md locked decision "sync immediately when user switches language").

**4. src/hooks/useProviders.ts -- Add refreshTrayMenu after CRUD operations**

IMPORTANT: The app's own writes to iCloud storage are recorded by `SelfWriteTracker` and filtered out by the file watcher. This means frontend CRUD operations (create, update, delete, switch, copy) will NOT automatically trigger tray rebuild via the watcher path. The solution: explicitly call `refreshTrayMenu()` after each state-changing operation.

Import `refreshTrayMenu` from `@/lib/tauri`. Then add `await refreshTrayMenu()` after each successful operation in the hook's callback functions: after create, update, delete, switch (setActiveProvider), and copy operations. Place the call AFTER the existing state updates (fetchProviders, refreshSettings) so the tray reads the latest data. Use fire-and-forget pattern (`refreshTrayMenu().catch(() => {})`) to avoid blocking the UI if the command fails.

**No changes needed to src/hooks/useSyncListener.ts** -- it already listens for `providers-changed` events. When the tray switches a provider (Plan 01 emits `providers-changed`), the existing listener calls `refreshProviders()` + `refreshSettings()`. This handles tray-to-frontend sync without modification.
