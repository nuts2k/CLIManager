    ```rust
    #[derive(Debug, serde::Serialize)]
    pub struct ProviderStat {
        pub provider_name: String,
        pub request_count: i64,
        pub success_count: i64,
        pub total_input_tokens: i64,
        pub total_output_tokens: i64,
        pub cache_triggered_count: i64,

    #[derive(Debug, serde::Serialize)]
    pub struct TimeStat {
        pub label: String,      // "HH:00" 或 "YYYY-MM-DD"
        pub request_count: i64,
        pub total_tokens: i64,  // input + output 合计
    }
    ```

    **rollup_and_prune 方法：** 实现为 `impl super::TrafficDb` 方法。

    使用 `ON CONFLICT(provider_name, rollup_date) DO UPDATE SET` 增量 upsert 语法（非 INSERT OR REPLACE），防止多次调用丢失累积数据。RESEARCH.md Pitfall 1 明确警告此问题。

    单次事务内三步：
    1. INSERT INTO daily_rollups ... ON CONFLICT DO UPDATE SET request_count = request_count + excluded.request_count, ...（聚合超 24h 明细）
    2. DELETE FROM request_logs WHERE created_at < (strftime('%s', 'now') - 86400) * 1000
    3. DELETE FROM daily_rollups WHERE rollup_date < strftime('%Y-%m-%d', 'now', '-7 days')

    注意 created_at 是 epoch 毫秒，SQL 中 strftime('%s','now') 返回秒，需乘以 1000 做阈值比较。RESEARCH.md Pitfall 2 明确警告此问题。

    **query_provider_stats 方法：** 接受 range 参数（"24h" / "7d"）。
    - "24h"：从 request_logs WHERE created_at >= 阈值 GROUP BY provider_name
    - "7d"：从 daily_rollups WHERE rollup_date >= 阈值 GROUP BY provider_name（需合并同 provider 多天数据）
    返回 Vec<ProviderStat>。

    **query_time_trend 方法：** 接受 range 参数。
    - "24h"：从 request_logs 按 strftime('%H:00', created_at/1000, 'unixepoch') 分组
    - "7d"：从 daily_rollups 按 rollup_date 分组
    返回 Vec<TimeStat>。

    **单元测试（#[cfg(test)] mod tests）：** 仿照 log.rs 的 make_test_db() 模式创建内存 DB。使用 make_test_db_with_logs() 辅助函数插入不同时间戳的测试数据。因为 rollup_and_prune 使用 strftime('%s', 'now')，测试中需插入**真实的超 24h 前时间戳**来验证。可以用 `(chrono::Utc::now().timestamp() - 86400 - 3600) * 1000` 作为超 24h 前的 created_at 值。

    在 mod.rs 中添加 `pub mod rollup;` 行（保持模块声明按字母排序：db, log, rollup, schema）。

       ```rust
       let app_handle_for_rollup = app.handle().clone();
       tauri::async_runtime::spawn(async move {
           use tauri::Manager;
           loop {
               if let Some(db) = app_handle_for_rollup.try_state::<crate::traffic::TrafficDb>() {
                   match db.rollup_and_prune() {
                       Ok(_) => log::info!("rollup_and_prune 执行完成"),
                       Err(e) => log::warn!("rollup_and_prune 执行失败: {}", e),
                   }
               }
               tokio::time::sleep(std::time::Duration::from_secs(3600)).await;
           }
       });
       ```
       放置位置：在 `tauri::async_runtime::spawn(log_worker)` 之后，Startup overlay apply 之前。

    **注意事项：**
    - 使用 `tauri::async_runtime::spawn`（非 tokio::spawn），RESEARCH.md Anti-Pattern 警告 Tauri 2 中 tokio::spawn 在 listener 内会 panic
    - rollup_and_prune 内部使用同步 Mutex（std::sync::Mutex），在 async spawn 内调用是安全的——因为持锁时间短且无跨 .await 持锁
    - 失败只 log::warn 不重试，等下一轮自动触发
